# 업무 협업툴 개선 방향성 - 프로젝트 스펙 (PHVI)

> 프레임워크: 문제정의 → 가설 → 검증 → 개선 (PHVI 사이클)

---

## 문제 정의 (Problem)

### P1: 비동기 피드백 지연
PM이 스프레드시트를 열어야만 팀원의 업무 상태를 확인할 수 있는 구조. 업무 완료 전까지 실시간 피드백이 불가능하다.
- **가설**: H1 상시 상주형 협업툴, H2 상태변경 즉시알림
- **세부문제**: 중간 피드백 부재(P4), 대면만 빠른 피드백(P5)

### P2: PM 병목 현상
모든 확인과 피드백이 PM 한 명에게 집중된다. PM의 스프레드시트 접속 타이밍에 전체 팀의 상황 파악이 의존하는 구조적 병목.
- **가설**: H2 상태변경 즉시알림

### P3: 수동 기록 의존
업무 생성(회의)→기록(시트)→완료표기→확인까지 전부 수동 프로세스. 누락과 지연이 구조적으로 발생.
- **세부문제**: 업데이트 습관 부재(P6)
- **가설**: H3 기록 동기부여 필요

### P4: 중간 피드백 부재
진행 중 막히거나 방향이 바뀌었을 때 시트에 메모를 남기지만, PM이 확인하기 전까지 사실상 방치. JANDI에서도 동일 문제 발생.

### P5: 대면만 빠른 피드백
급한 업무만 사무실에서 직접 대면하여 해결. 일반적인 업무는 구조적으로 피드백이 느릴 수밖에 없는 환경.

### P6: 업데이트 습관 부재
팀원들이 바쁨, 귀찮음, 손이 안 감 등의 이유로 상태 업데이트를 미루거나 빠뜨림. 수동 기록 시스템의 구조적 한계.

---

## 가설 (Hypothesis)

### H1: 상시 상주형 협업툴
> "Sticky Notes처럼 데스크탑에 항상 떠 있는 협업툴을 만들면, PM이 별도 앱을 열지 않아도 업무 상태 변화를 즉시 인지할 수 있을 것이다."
- **문제**: P1 비동기 피드백 지연
- **핵심 전제**: 팀원들이 상태를 제때 업데이트할 것
- **세부 가설**: H1-1 작고 항상 보이는 UI
- **검증**: V1 MVP 직접 개발 필요

### H2: 상태변경 즉시알림
> "업무 상태 변경 시 사운드/비주얼 알림이 자동 발생하면, PM 병목 없이 실시간 피드백 루프가 만들어질 것이다."
- **문제**: P2 PM 병목 현상

### H3: 기록 동기부여 필요
> "업무 기록 자체에 동기(인센티브)를 부여하면, 팀원들이 자발적으로 상태를 업데이트할 것이다."
- **문제**: P6 업데이트 습관 부재
- **검증**: V2 2주 보상 실험
- **세부 가설**: H4 게이미피케이션 필수

### H4: 게이미피케이션 필수
> "게이미피케이션(포인트/배지/랭킹)을 협업툴에 내장하면, 외부 보상 없이도 자발적 기록 동기가 지속될 것이다."
- **세부 가설**: H4-1 포인트/배지/랭킹 → H4-2 아바타 커스터마이징, H4-3 미니게임 포인트 소모
- **검증**: V5 디지털→실물 보상 연결

### H5: 진입 비용 제로 설계
> "별도로 '들어가야 하는' 행위를 제거하고 항상 업무 영역에 머물러 있는 위젯 형태로 만들면 도구 피로감을 극복할 수 있을 것이다."
- **위험 대응**: 새 툴 도입 피로감
- **세부 가설**: H5-1 이벤트 기반 자연 접근, H5-2 사운드/비주얼 알림

---

## 검증 계획 (Validation)

### V1: MVP 직접 개발 필요
상시 상주형 협업툴은 기존 도구로 대체 불가. 최소 기능의 프로토타입을 직접 만들어야 한다.

### V2: 2주 보상 실험
PM이 업무 현황 체크를 최우선으로 수행하고, 부서별 업무 공유도에 따른 보상(커피)을 제시. 1~2주간 동기부여 효과를 테스트.
- **지표 1**: 스프레드시트 업데이트 횟수 (실험 전후 비교)
- **지표 2**: 업무 상태 변경→PM 피드백 소요시간 측정

### V5: 디지털→실물 보상 연결
디지털 보상(아바타)만으로 충분한지, 실물 보상(커피권) 연결이 필수인지를 실험으로 분리 측정.

---

## 개선 방향 (Improvement)

### I1: PM 과부하 진단
피드백 소요시간이 줄지 않으면 PM 업무 과부하를 의심. PM의 업무량 자체를 점검하고 조정.
- **후속**: PM 업무조정 + 보상안 설계 (양방향 개선)

### I2: A→B 순차 검증
1단계: 보상 실험으로 동기부여 효과 검증 → 2단계: 결과 반영해 상시 상주형 협업툴 요구사항 구체화 후 MVP 개발.

### I3: PM 승인 후 포인트
완료 표기만으로 포인트 지급하지 않음. PM이 업무 품질을 확인한 뒤 최종 승인 시 포인트 적립. 어뷰징 방지.

### I4: 실물 교환 상점
포인트로 실물 보상(커피 교환권 등) 구매 가능. 디지털+실물 보상이 공존하여 동기부여 지속성 확보.

---

## 대상 사용자

- **소규모 스타트업 (8명)**: PM 1명 + 운영4 + 기획1 + 디자인2
- **협의 기반 우선순위**: 팀 리드-팀원 간 협의로 결정, 긴급 시 리드 독단
- **회의록 기반 전달**: 리더회의→팀회의 2단계 구조는 잘 작동 중 (문제 없음)

## 경쟁 분석

- **JANDI**: 기존 사용 협업툴. PM이 확인해야 피드백 가능한 동일 문제 반복. 도구가 아닌 프로세스가 문제라는 교훈.

## 리스크

- **새 툴 도입 피로감**: JANDI→스프레드시트 전환 경험으로 피로 누적. 진입 비용 제로 설계로 대응.
- **그래픽 리소스 필요**: 아바타 파츠, 상점 아이템 등 제작 비용. MVP 범위 제한 필요.

## 기술

- **데스크탑 앱 (Tauri)**: Rust 기반 경량 프레임워크. Windows Sticky Notes 같은 가벼운 UX 목표.
- **AI 협업 개발**: 기술 지식 없이 기획자가 설계하고 AI가 구현하는 방식.
